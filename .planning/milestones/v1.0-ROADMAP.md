# Milestone v1.0: Hardening

**Status:** ✅ SHIPPED 2026-04-20
**Phases:** 1–3
**Total Plans:** 10

## Overview

Hardened the existing Firefox WebExtension (Manifest V3) for O'Reilly EPUB downloads. Three phases attacked the problem in priority order: performance and worker reliability first (most user-visible), then code-correctness bugs causing silent failures, then security hardening closing injection and message-spoofing vectors.

## Phases

### Phase 1: Performance & Worker Reliability

**Goal**: Large book downloads complete without exhausting memory, and in-flight downloads survive Manifest V3 worker restarts
**Depends on**: Nothing (first phase)
**Plans**: 3 plans

Plans:
- [x] 01-01-PLAN.md — Add getCachedFiles(ourn) to BookCache (PERF-03 foundation)
- [x] 01-02-PLAN.md — Refactor download.js: JSZip threading, per-file compression, session write-through (PERF-01, PERF-02, PERF-03, REL-01 sendProgress)
- [x] 01-03-PLAN.md — Startup restore from session storage + manifest version bump (REL-01 background, manifest)

**Details:**
- Success criteria: 500+ file downloads stay memory-bounded; pre-compressed images use STORE mode; single IndexedDB cursor pass; worker restart recovery via browser.storage.session

### Phase 2: Code Correctness

**Goal**: Downloads complete correctly and cleanly — no resource leaks, no encoding failures, no broken modes, no dead message handlers
**Depends on**: Phase 1
**Plans**: 5 plans

Plans:
- [x] 02-01-PLAN.md — URL-decode JWT cookie value in content.js getJWTToken() (BUG-03)
- [x] 02-02-PLAN.md — saveFile try/finally blob URL revocation + non-recursive OURN fallback (BUG-01, REL-03)
- [x] 02-05-PLAN.md — Add storeFileManifest/getFileManifest to BookCache in cache.js (BUG-04 persistence layer)

**Details:**

### Phase 3: Security Hardening

**Goal**: The extension does not process messages from unknown senders, does not produce XSS-vulnerable output, and does not corrupt EPUB XML with unescaped characters
**Depends on**: Phase 2
**Plans**: 2 plans

Plans:
- [x] 03-01-PLAN.md — Insert sender.id guard in onMessage listener (SEC-01)
- [x] 03-02-PLAN.md — DOMParser cleanHTML rewrite + preserve link elements + escapeXml helper (SEC-02, BUG-02, SEC-03)

**Details:**
- Success criteria: Unknown senders rejected; DOMParser replaces regex; link tags preserved; XML escaping applied; ZIP path traversal sanitized

## Milestone Summary

**Key Decisions:**

- Performance before security — user-prioritized; memory issues more immediately impactful ✓ Good
- DOMParser over regex for HTML sanitization — available in service workers; structurally sound ✓ Good
- browser.storage.session for activeDownloads — survives brief worker restarts without persisting across browser restarts ✓ Good
- Include bugs and hardening together — interleaved; fixing together more efficient ✓ Good

**Issues Resolved:**

- Memory bloat on large book downloads (500+ files)
- Object URL leaks on download failure
- JWT URL-encoding bug causing 401 errors
- Rebuild from Cache mode network leak
- sendProgress global coupling
- Dead GET_BOOK_INFO handler
- Script-stripping regex removing CSS link tags
- XML injection in generateContainerXml
- Message spoofing from external extensions/pages
- ZIP path traversal via API-supplied paths
- Pagination next URL without host validation
- activeDownloads.set(undefined) collision on missing ourn

**Issues Deferred:**

- Human verification of memory bounds and worker restart recovery (requires live Firefox session)
- popup shows "Build from Cache" after worker restart mid-download — should show "interrupted, retry?"
- sendProgress fire-and-forget missing .catch(() => {}) at download.js:379

**Technical Debt Incurred:**

- No automated test suite — all validation is manual (TEST-v2-01, TEST-v2-02 deferred to next milestone)
- JSZip version untracked — not pinned against published hash (SEC-v2-02 deferred)
